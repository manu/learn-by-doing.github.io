# Exercise 3.8 — .gitignore: What Not to Track

## The Story

A developer commits their project to GitHub. A week later, they get an email from a security researcher:

> "Your AWS access keys are public in your repository. Someone has already used them. Your bill for this month is $47,000."

The developer had committed a file called `.env` containing their secret keys. They never meant to — it was just sitting in the folder. They used `git add .` and it went in.

This is not rare. It happens to experienced engineers. GitHub has automated scanners that find leaked credentials within seconds of a push.

`.gitignore` is the file that tells Git which files to never track. It is one of the first things you create in any project.

## What Never Belongs in Version Control

There are four categories of files that should never be committed:

**Secrets and credentials**
- `.env` files containing API keys, passwords, database URLs
- `config.json` or `settings.py` with secret values
- SSH private keys

**Generated and compiled files**
- `__pycache__/` and `.pyc` files — Python generates these automatically
- Build outputs, compiled binaries
- Files generated by running the program

**Data files that belong to users, not the project**
- `library.csv` — this is the library's data, not the code
- Databases, log files, uploaded files

**Editor and operating system noise**
- `.DS_Store` (macOS folder metadata)
- `Thumbs.db` (Windows thumbnail cache)
- `.vscode/`, `.idea/` (editor settings that only apply to your machine)

## What to Do

### Part A — Audit what is currently tracked

Run `git status` and `git ls-files` in your Mini Library. Look at every file Git is tracking. For each one, answer: should this file be in version control? Why or why not?

Write down every file that should NOT be tracked.

### Part B — Create a .gitignore

Create a file called `.gitignore` in the root of your project. Add rules for:

```
# Data files
library.csv

# Python generated files
__pycache__/
*.pyc
*.pyo

# Environment and secrets
.env
*.env

# Operating system files
.DS_Store
Thumbs.db

# Editor files
.vscode/
.idea/
*.swp
```

Test it: run `git status`. The files matching the rules should no longer appear as untracked.

### Part C — Commit the .gitignore

```
git add .gitignore
git commit -m "Add .gitignore to exclude data files, caches, and OS noise"
```

### Part D — The harder case: a file already tracked

What if you have already committed `library.csv`? Adding it to `.gitignore` does NOT remove it from history — it only stops future tracking.

To stop tracking an already-committed file:
```
git rm --cached library.csv
git commit -m "Remove library.csv from tracking (data file, not source code)"
```

The file stays on your disk — `--cached` means "remove from Git, not from the filesystem." After this, `.gitignore` will prevent it from being re-added.

### Part E — The secret question

Why should secrets never be committed — even privately, even "just once," even if you delete the commit later?

Research: can you remove a file from Git history entirely? Look up `git filter-branch` or the BFG Repo Cleaner. What does this tell you about how permanent git history is?

Write two sentences: what you found and why this means "never commit secrets" is a stronger rule than "delete secrets after committing."

## Topics You Will Learn

- What `.gitignore` is and how its pattern syntax works (`*`, `/`, `!`)
- The four categories of files that should never be committed
- How to stop tracking a file that was accidentally committed
- Why committed secrets cannot be truly erased and must be rotated

## Before You Start — Think About This

1. `git add .` adds every file in the folder. Why is this dangerous without a `.gitignore`? What is the safer alternative?
2. The `.gitignore` file itself IS committed to the repository. Why does it make sense to version-control the ignore rules?
3. If your `.gitignore` excludes `library.csv`, and someone else clones your repository, what do they get? What do they need to do to run the program?

## When You're Stuck

- `.gitignore` patterns: `*.csv` ignores all CSV files; `data/*.csv` ignores CSV files only inside the `data/` folder; `!important.csv` is an exception (don't ignore this specific file even if `*.csv` matches it).
- If a file is already tracked and in `.gitignore`, Git still tracks it — `.gitignore` only affects untracked files. Use `git rm --cached filename` to fix this.
- GitHub provides ready-made `.gitignore` templates for Python when you create a new repository. Use them as a starting point.

## Once It Works — Go Further

1. Go to `github.com/github/gitignore` — GitHub's collection of `.gitignore` templates. Find the Python template. What does it include that your `.gitignore` does not? Add anything relevant.
2. Look up what a `.env` file is and why it is the standard way to store local secrets. How would you let other developers know that they need to create a `.env` file without putting the actual secrets in the repository? (Look up `.env.example`.)
